Raft KV — consensus, from the paper up

A distributed key-value store where multiple servers agree on data even when nodes crash — every line of the consensus layer written by hand in Go.

The problem

How do multiple servers agree on the same data when any of them can crash, restart, or get partitioned? Consensus is the foundation under etcd, Consul, and CockroachDB — and reading the Raft paper is not the same as making it work. I wanted the version of understanding that only comes from debugging your own leader elections at 2am.

What I built

Leader election with randomized timeouts, straight from the paper Log replication with consistency checks and conflict resolution Snapshots so restarting nodes don't replay the whole log Linearizable reads — reads reflect all committed writes Dynamic cluster membership — add and remove nodes live

Engineering decisions

The hardest part wasn't the happy path — it was the edges: a stale leader that doesn't know it's stale, a follower whose log diverged during a partition, election storms when timeouts are too tight. Key choices:

No consensus libraries. The point was understanding — using etcd's raft package would defeat it. State machine as a pure function. Applying log entries is deterministic and side-effect free, which makes replay and snapshots trivial to reason about. Chaos-style testing. Tests kill leaders mid-replication and partition nodes randomly — most bugs only appear under induced failure.

What it taught me

Distributed systems fail in ways sequential code never prepares you for. Building Raft gave me a working mental model for every consensus-backed system I touch now — and it's the foundation the rest of my systems work (like Flume) builds on.
